# Security

> **Warning.**
>
> Безопасность закладывается на этапе написания кода, а не тестирования.

## **Аутентификация и авторизация**

Каждый endpoint должен быть явно защищён и требовать аутентификацию и/или авторизацию.

Принцип:

**"Запрещено всё, что не разрешено явно"**

В конфигурации Spring Security обязательно должно быть:

```
.anyRequest().authenticated()
```

Это гарантирует, что:

- любой новый endpoint по умолчанию закрыт
- разработчик не сможет случайно открыть доступ к API
- отсутствуют "дыры" при расширении системы

Все публичные endpoints должны задаваться явно и точечно:

```
.requestMatchers(
 	"/api/v1/auth/login",
 	"/api/v1/auth/refresh",
 	"/actuator/health"
).permitAll()
```

В production конфигурации запрещено использовать глобальные разрешения:

```
.anyRequest().permitAll()
```

## **Проверка прав доступа**

Наличие аутентификации (валидного токена) не означает наличие доступа. Каждый защищённый метод должен проверять роли или permissions.

Пример:

```
@PreAuthorize("hasAuthority('WRITE_PRIVILEGE')")
```

Основные требования:

- Нельзя оставлять endpoint без явных правил доступа
- Нельзя полагаться только на frontend или gateway
- Все проверки безопасности должны выполняться строго на backend уровне
- Любой новый endpoint должен сопровождаться настройкой доступа

## **Проверка доступа к данным пользователя (Insecure Direct Object Reference (IDOR))**

В каждом backend-сервисе на Java необходимо явно проверять, что пользователь имеет право запрашивать или изменять конкретные данные. Пользователь может работать только со своими данными, если не предусмотрены особые роли (например, администратор).

Любой запрос, содержащий userId, accountId, orderId и т.п., не должен слепо доверяться входящим данным. Вместо этого необходимо:

- получать идентификатор пользователя из аутентифицированного контекста (например, JWT, SecurityContext)
- сравнивать его с владельцем запрашиваемого ресурса
- запрещать доступ при несовпадении

**Как это реализуется:**

1) Получение текущего пользователя

```
String currentUserId = securityContext.getAuthenticatedUserId();
```

2) Загрузка ресурса

```
Order order = orderRepository.findById(orderId)
.orElseThrow(() -> new NotFoundException("Order not found"));
```

3) Проверка владельца

```
if (!order.getUserId().equals(currentUserId)) {
    throw new AccessDeniedException("Access denied");
}
```

**Где должна находиться проверка**

- В service-слое (обязательно)
- Дополнительно можно дублировать на уровне контроллеров (для раннего отказа) или базы данных

**Антипаттерны**

- Доверие userId из request body / query params
- Отсутствие проверки при работе с ID ресурсов
- Проверка только на уровне UI (frontend) без backend-валидации

**Рекомендуемые практики**

- Использовать Spring Security или аналогичный механизм
- Централизовать проверки через:
- AOP (аспекты)
- аннотации (@PreAuthorize)
- Поддерживать модель ролей, если есть разные уровни доступа
- Логировать попытки несанкционированного доступа

**Пример с аннотацией**

```
@PreAuthorize("@securityService.isOwner(#orderId, authentication)")
public Order getOrder(String orderId) {
    return orderService.getOrder(orderId);
}
```

Этот пункт обязателен для всех API, работающих с пользовательскими данными, и защищает от Insecure Direct Object Reference (IDOR).

## **Управление секретами**

Категорически запрещено:

- хардкодить пароли
- хранить токены в коде
- хранить секреты в `application.yml`

Необходимо использовать:

- переменные окружения
- Vault
- Kubernetes secrets

## **Безопасность логирования**

Никогда не следует записывать в логи данные, которые могут быть использованы для компрометации системы или нарушения приватности пользователя.

Плохо:

Логирование секретов в открытом виде. При утечке файлов логов злоумышленник получает полный доступ к аккаунтам.

```
log.info("User login attempt: login={}, password={}", login, password);
```

Хорошо:

- Маскирование или удаление: оставляйте только факт события и нечувствительные идентификаторы.
- Хеширование: eсли нужно отследить путь конкретного пользователя, не раскрывая его, логируйте id или хеш от логина.

```
log.info("User login attempt for username: {}", login);

log.info("Processing payment for userId: {}, orderId: {}", user.getId(), order.getId());
```

Что нельзя логировать:

- Пароли, пин-коды, секретные вопросы
- Токены доступа (JWT, API keys) и сессии
- Персональные данные в связке (ФИО + номер телефона/паспорт)
