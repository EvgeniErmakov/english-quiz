# Architecture and Code Structure

## Структура кода

### Слоистая архитектура

Необходимо придерживаться строгого разделения слоёв: **Controller → Service → Repository → Database**

- **Controller** — не должен содержать бизнес-логику. Отвечает за получение HTTP-запроса, валидацию входных данных, вызов сервисного слоя и возврат ответа
- **Service** — содержит всю бизнес-логику
- **Repository** — отвечает только за работу с базой данных

Рассмотрим путь данных используемый в проекте:

**Client ➔ Controller ➔ DTO ➔ Service ➔ Mapper ➔ Domain/Entity ➔ Repository ➔ Database**

1. **Client:** Отправляет HTTP POST запрос с JSON-телом на сервер.
2. **Controller:** Принимает запрос, десериализует JSON в объект. Проверяет базовую валидацию (например, `@Valid`). Вызывает метод сервиса.
3. **Service:** Получает `DTO`. Здесь выполняется бизнес-логика. Сервис вызывает `Mapper`.
4. **Mapper:** Преобразует (маппит) данные из Dto в сущность Entity.
5. **Repository:** Сервис передает сущность Entity в репозиторий. Репозиторий через Hibernate/JPA генерирует SQL-запрос.
6. **Database:** Сохраняет запись в таблицу и возвращает сгенерированный ID.
7. **Обратный путь:** Репозиторий возвращает сохраненную сущность `Entity` в сервис. Сервис использует `Mapper`, чтобы превратить Entity в Dto.
8. **Response:** `Controller` оборачивает `Dto` в `ResponseEntity` и отправляет клиенту со статусом `201 Created`.

### Структура проекта

В проекте должна использоваться единая и предсказуемая структура директорий с логической группировкой компонентов.

Рассмотрим пример многослойной архитектуры (Layered Architecture), которая помогает четко разделить ответственности компонентов:

```
 src/main/java/ru/lanit/ppoz/outage 
├── controller 
├── service 
│ └── impl
├── repository
├── domain
├── dto 
├── enums 
├── mapper 
├── config 
├── exception 
├── security 
├── messaging 
├── util 
├── validation 
```

**1. `controller`**

За что отвечает: Входящие HTTP-запросы.

- Принимает данные от клиента (JSON, параметры запроса).
- Вызывает методы сервисного слоя.
- Возвращает HTTP-ответы (`ResponseEntity`).

**2. `service` & `service.impl`**

За что отвечает: Бизнес-логика приложения.

- `service` (интерфейсы): Определяет контракт (какие действия возможны).
- `service.impl` (реализации): Содержит саму логику, расчеты, вызовы репозиториев.
- Здесь происходит управление транзакциями (`@Transactional`).

**3. `repository`**

За что отвечает: Доступ к данным (Data Access Layer).

- Интерфейсы, расширяющие `JpaRepository`, `CrudRepository`.
- Содержит SQL или JPQL запросы.

**4. `domain` (или `entity` / `model`)**

За что отвечает: Сущности базы данных.

- Классы, помеченные аннотацией `@Entity`.
- Описывают структуру таблиц в БД.

**5. `dto` (Data Transfer Object)**

За что отвечает: Обмен данными между слоями и клиентом.

- Объекты, которые передаются в Controller или возвращаются из него.
- Помогают скрыть внутреннюю структуру `domain` моделей от внешнего мира.

**6. `mapper`**

За что отвечает: Преобразование объектов.

- Логика конвертации `Entity` в `DTO` и обратно.
- Часто используются библиотеки вроде MapStruct.

**7. `config`**

За что отвечает: Настройки приложения.

- Классы с аннотацией `@Configuration`.
- Настройка бинов, Swagger/OpenAPI, kafka, кэширования и т.д.

**8. `security`**

За что отвечает: Авторизация и аутентификация.

- Конфигурация Spring Security.
- JWT фильтры, кастомные `UserDetailsService`.

**9. `exception`**

За что отвечает: Обработка ошибок.

- Кастомные классы исключений (наследуются от `RuntimeException`).
- `GlobalExceptionHandler` (с аннотацией `@ControllerAdvice`) для перехвата ошибок во всем приложении.

**10. `enums`**

За что отвечает: Перечисления.

- Константы, которые используются в логике или БД.

**11. `messaging`**

За что отвечает: Взаимодействие с очередями сообщений.

- Продюсеры (Producers) и Слушатели (Listeners/Consumers) для Kafka, RabbitMQ.

**12. `util`**

**З**а что отвечает: Вспомогательные инструменты.

- Статические методы, которые не несут бизнес-логики, но полезны везде.
- Работа с датами, строками, шифрованием.

**13. `validation`**

За что отвечает: Кастомная валидация.

- Аннотации для специфических проверок и их реализации (`ConstraintValidator`).
