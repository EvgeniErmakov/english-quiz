# API Guidelines

## **REST-конвенции**

API является контрактом между сервисами и клиентами. Ошибки в дизайне API приводят к:

- сложной поддержке
- росту технического долга

Поэтому API должно проектироваться осознанно и единообразно, соответствовать REST-принципам и быть предсказуемым, консистентным и понятным.

Требования к именованию ресурсов:

- используются существительные, а не глаголы
- ресурсы именуются во множественном числе

Правильно:

```
GET /users
GET /users/{id}
POST /users
```

Неправильно:

```
GET /getUsers
POST /createUser
```

Иерархия ресурсов

Связанные ресурсы должны отражаться в URL:

```
GET /users/{userId}/orders
GET /orders/{orderId}/items
```

### Фильтрация, сортировка, пагинация

Фильтрацию всегда следует выполнять через query-параметры:

```
GET /users?status=ACTIVE&role=ADMIN
```

Сортировка:

```
GET /users?sort=createdAt,desc
```

Пагинация. Все list endpoints обязаны поддерживать пагинацию, пример:

```
GET /users?page=0&size=10
```
> **Warning.**
>
> Нельзя создавать endpoints, которые возвращают неограниченный список данных.

## **HTTP-методы и статусы**

Каждый HTTP метод должен использоваться строго по назначению:

| Метод | Действие | Описание |
| --- | --- | --- |
| GET | Получение | Запрашивает данные с сервера |
| POST | Создание | Создаёт новый ресурс |
| PUT | Обновление | Полностью заменяет существующий ресурс |
| PATCH | Правка | Частично изменяет ресурс |
| DELETE | Удаление | Удаляет указанный ресурс |

В свою очередь каждый endpoint должен возвращать корректный HTTP статус.

Нельзя всегда возвращать `200 OK`.

| Статус | Когда использовать |
| --- | --- |
| 200 OK | успешный GET / PUT |
| 201 Created | ресурс успешно создан |
| 204 No Content | ресурс успешно удалён |
| 400 Bad Request | ошибка валидации запроса |
| 401 Unauthorized | пользователь не аутентифицирован |
| 403 Forbidden | у пользователя нет прав |
| 404 Not Found | ресурс не найден |
| 405 Method Not Allowed | HTTP метод не поддерживается |
| 500 Internal Server Error | внутренняя ошибка сервера |
| 503 Service Unavailable | сервис временно недоступен |

## **Обработка ошибок**

В контроллерах не должно быть блоков `try-catch` для обработки бизнес-ошибок, для этого необходимо использовать глобальный обработчик исключений:

```
@RestControllerAdvice
```
