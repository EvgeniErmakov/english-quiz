# Code Quality

### Code Review

Каждый Pull Request **обязан** пройти Code Review.

**Минимальные требования:**

- успешный CI pipeline: проект собрался, все тесты пройдены успешно
- отсутствуют merge-конфликты
- минимум 1 approve от другого разработчика

**Основные цели Code Review:**

- выявление ошибок
- повышение качества архитектуры
- распространение знаний в команде
- предотвращение security проблем

### Тестирование

Тестирование является обязательной частью разработки и служит для предотвращения багов, повышения стабильности системы и упрощения рефакторинга. Цель тестов - гарантировать корректность бизнес-логики и предотвратить появление ошибок при изменении кода.

#### Unit-тесты

Unit-тесты должны покрывать бизнес-логику сервисного слоя.

В unit-тестах:

- зависимости мокируются (Mockito)
- тестируется только логика метода
- не используется реальная база данных

Unit-тесты позволяют:

- быстро находить ошибки в бизнес-логике
- безопасно проводить рефакторинг
- гарантировать корректность алгоритмов

#### Integration-тесты

Integration-тесты проверяют корректность взаимодействия приложения с инфраструктурой, прежде всего с базой данных.

Для integration-тестов используется **Testcontainers + реальный PostgreSQL**, что позволяет максимально приблизить тестовую среду к production.

Integration-тесты должны проверять:

- корректность SQL-запросов
- работу репозиториев
- корректность маппинга entity ↔ database
- работу транзакций

#### Тесты контроллеров

Контроллерные тесты проверяют корректность работы REST API.

Такие тесты должны проверять:

- корректность HTTP статусов
- валидацию входных данных
- структуру ответа API

Покрытие - не самоцель, но критическая логика должна быть покрыта. Например, к такой логике можно отнести:

**Форматный контроль** - проверка корректности входных данных перед созданием сущности. Например:

- формат даты
- обязательные поля
- допустимые значения параметров

Такие проверки предотвращают попадание некорректных данных в систему.

**Логический контроль** - проверка бизнес-правил системы. Например:

- нельзя создать объект с конфликтующими параметрами
- нельзя создать дубликат
- нельзя изменить объект в недопустимом состоянии

Тесты должны гарантировать, что система не позволит нарушить бизнес-правила.

**Обогащение данных** -дополнение входных данных дополнительной информацией:

- данными из базы
- подстановка параметров из справочников
- вычисляемыми значениями
- вычисление агрегированных значений

Такая логика должна быть протестирована, чтобы гарантировать корректность формирования итоговых данных.
